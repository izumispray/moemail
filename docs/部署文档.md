## 概述

MoeMail 使用 GitHub Actions 实现全自动 CI/CD。只需在 GitHub 仓库中配置好 Secrets，推送 Tag 或手动触发即可完成全栈部署。

**部署触发方式：**
- 推送 `v*` 格式的 Git Tag（如 `v1.0.0`）
- 在 GitHub Actions 页面手动点击 `Run workflow`

---

## 架构说明

MoeMail 由以下 Cloudflare 组件构成，全部通过一个部署脚本自动创建和部署：

```
┌─────────────────────────────────────────────────────────────┐
│                      Cloudflare 平台                         │
│                                                              │
│   ┌──────────────┐  ┌───────────────┐  ┌────────────────┐   │
│   │  Pages App   │  │ Email Worker  │  │  DNS Worker    │   │
│   │  (Next.js)   │  │  (接收邮件)    │  │  (DNS+路由)    │   │
│   └──────┬───────┘  └──────┬────────┘  └───────┬────────┘   │
│          │                 │                    │             │
│   ┌──────┴─────────────────┴────────────────────┘            │
│   │                                                          │
│   │   ┌──────────┐  ┌──────────────┐  ┌────────────────┐   │
│   │   │ D1 数据库 │  │ KV 键值存储   │  │ Cleanup Worker │   │
│   │   └──────────┘  └──────────────┘  └────────────────┘   │
│   │                                                          │
└───┴──────────────────────────────────────────────────────────┘
```

| 组件 | 说明 |
|------|------|
| **Pages App** | Next.js 主应用，提供 Web UI 和 REST API |
| **Email Worker** | 接收 Cloudflare Email Routing 转发的邮件，存入 D1 |
| **DNS Worker** | 管理子域名的 DNS 记录、启用 Email Routing、设置 catch-all 规则 |
| **Cleanup Worker** | 定时清理过期邮箱和消息 |
| **D1** | SQLite 数据库，存储用户、邮箱、消息等核心数据 |
| **KV** | 键值存储，保存域名列表、角色配置等站点设置 |

## 前置准备

### 1. Cloudflare 账号

注册免费的 [Cloudflare](https://dash.cloudflare.com) 账号，获取以下信息：

- **Account ID**：在 Cloudflare Dashboard 首页右侧栏获取
- **API Token**：在 [API Tokens](https://dash.cloudflare.com/profile/api-tokens) 页面创建（权限要求见下方）

### 2. OAuth 应用（至少配置一个）

#### GitHub OAuth

1. 前往 [GitHub Developer Settings](https://github.com/settings/developers)
2. 创建 New OAuth App
3. **Authorization callback URL** 设为：`https://你的域名/api/auth/callback/github`
4. 记录 Client ID 和 Client Secret

#### Google OAuth

1. 前往 [Google Cloud Console](https://console.cloud.google.com/apis/credentials)
2. 创建 OAuth 2.0 客户端
3. **授权重定向 URI** 设为：`https://你的域名/api/auth/callback/google`
4. 记录 Client ID 和 Client Secret

### 3. 域名（可选）

如果要使用自定义域名，需先将域名添加到 Cloudflare 并完成 DNS 托管。

## 配置 GitHub Secrets

在 GitHub 仓库 → **Settings** → **Secrets and variables** → **Actions** 中添加以下 Secrets：

### 必填 Secrets

| Secret 名 | 说明 | 获取方式 |
|-----------|------|---------|
| `CLOUDFLARE_ACCOUNT_ID` | Cloudflare 账户 ID | Dashboard 首页右侧栏 |
| `CLOUDFLARE_API_TOKEN` | Cloudflare API Token | [API Tokens 页面](https://dash.cloudflare.com/profile/api-tokens) |
| `AUTH_SECRET` | NextAuth 加密密钥 | 运行 `openssl rand -hex 32` 生成 |

### OAuth Secrets（至少配置一组）

| Secret 名 | 说明 |
|-----------|------|
| `AUTH_GITHUB_ID` | GitHub OAuth Client ID |
| `AUTH_GITHUB_SECRET` | GitHub OAuth Client Secret |
| `AUTH_GOOGLE_ID` | Google OAuth Client ID |
| `AUTH_GOOGLE_SECRET` | Google OAuth Client Secret |

### 可选 Secrets

| Secret 名 | 说明 | 默认值 |
|-----------|------|--------|
| `PROJECT_NAME` | 项目名称前缀 | `moemail` |
| `DATABASE_NAME` | D1 数据库名 | `moemail-db` |
| `DATABASE_ID` | 已有数据库 ID（跳过创建） | 自动创建 |
| `KV_NAMESPACE_NAME` | KV 命名空间名 | `moemail-kv` |
| `KV_NAMESPACE_ID` | 已有 KV ID（跳过创建） | 自动创建 |
| `CUSTOM_DOMAIN` | 自定义域名 | Pages 默认域名 |
| `DNS_WORKER_SECRET` | DNS Worker 密钥 | **自动生成** |
| `DNS_WORKER_URL` | DNS Worker 地址 | **自动获取** |

> 💡 `DNS_WORKER_SECRET` 和 `DNS_WORKER_URL` 在首次部署时会自动生成。如果你想固定这些值（避免每次部署生成新密钥），可以在首次部署成功后将它们保存到 Secrets 中。

## 部署流程详解

部署脚本 (`scripts/deploy/index.ts`) 按顺序自动执行以下步骤：

```
步骤 1   验证环境变量
  ↓      检查 CLOUDFLARE_ACCOUNT_ID 和 CLOUDFLARE_API_TOKEN
步骤 2   创建 .env 文件
  ↓      从 .env.example 模板生成，填入 Secrets
步骤 3   生成 Wrangler 配置
  ↓      从 *.example.json 模板生成 wrangler.json 等
步骤 4   创建/验证 D1 数据库
  ↓      不存在则自动创建
步骤 5   执行数据库迁移
  ↓      应用 drizzle/ 目录下的 SQL 迁移
步骤 6   创建/验证 KV 命名空间
  ↓      不存在则自动创建
步骤 7   创建/验证 Pages 项目
  ↓      不存在则自动创建
步骤 8   部署 DNS Worker
  ↓      部署 Worker + 注入 Secrets：
  ↓        • CLOUDFLARE_API_TOKEN
  ↓        • DNS_WORKER_SECRET（自动生成）
  ↓        • EMAIL_WORKER_NAME（自动设置）
  ↓      自动获取 Worker URL 并写入 .env
步骤 9   推送 Secrets 到 Pages
  ↓      将 .env 中的运行时 Secrets 推送到 Pages
步骤 10  部署 Pages 应用
  ↓      构建 Next.js 并发布
步骤 11  部署 Email Worker
  ↓      部署邮件接收 Worker
步骤 12  部署 Cleanup Worker
  ↓      部署定时清理 Worker
步骤 13  清理临时文件
         删除 .env 和 wrangler.json 等敏感文件
```

### 推送到 Pages 的运行时 Secrets

以下 Secrets 会从 `.env` 中读取并推送给 Pages 应用：

| Secret | 用途 |
|--------|------|
| `AUTH_GITHUB_ID` | GitHub 登录 |
| `AUTH_GITHUB_SECRET` | GitHub 登录 |
| `AUTH_GOOGLE_ID` | Google 登录 |
| `AUTH_GOOGLE_SECRET` | Google 登录 |
| `AUTH_SECRET` | Session 加密 |
| `CLOUDFLARE_API_TOKEN` | 调用 CF API（域名管理） |
| `DNS_WORKER_URL` | 调用 DNS Worker |
| `DNS_WORKER_SECRET` | 认证 DNS Worker |

## Cloudflare API Token 权限

创建 API Token 时需要以下权限：

| 权限 | 级别 | 用途 |
|------|------|------|
| **Account > D1** | Edit | 创建和管理数据库 |
| **Account > Workers KV Storage** | Edit | 创建和管理 KV 命名空间 |
| **Account > Workers Scripts** | Edit | 部署 Workers |
| **Account > Cloudflare Pages** | Edit | 部署 Pages 应用 |
| **Zone > DNS** | Edit | 创建 MX / SPF 记录 |
| **Zone > Email Routing Rules** | Edit | 启用 Email Routing、设置 catch-all 规则 |

> ⚠️ **Zone > Email Routing Rules** 是子域名邮箱自动化的关键权限。缺少此权限时 DNS 记录可以创建，但邮件路由不会生效。

### 创建步骤

1. 打开 [Cloudflare API Tokens](https://dash.cloudflare.com/profile/api-tokens)
2. 点击 **Create Token**
3. 选择 **Create Custom Token**
4. 按上表添加权限
5. Zone Resources 选择 **All zones**（或指定域名）
6. 创建后复制 Token 值

## 首次部署后初始化

部署成功后，需要完成以下初始化：

### 1. 初始化皇帝角色

首次使用时，第一个登录的用户需要初始化为皇帝：

```bash
# 使用浏览器登录后访问
https://你的域名/api/roles/init-emperor
```

### 2. 配置邮箱域名

登录后进入**管理面板**：

1. 在 **邮箱域名** 中添加你的域名（如 `example.com`）
2. 在 **域名 Zone 映射** 中填写域名对应的 Cloudflare Zone ID

> Zone ID 在 Cloudflare Dashboard 中对应域名的概览页右下角获取。

### 3. 创建 API Key（可选）

如果需要通过 API 调用，在**设置**中创建 API Key：

- 点击 **创建 API Key**
- 为 Key 命名
- 保存返回的 `mk_xxxx` 格式密钥

后续 API 请求在 Header 中携带：
```
X-API-Key: mk_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

### 4. 配置 Email Routing（首次）

对于第一次使用的域名，需在 Cloudflare Dashboard 中确认 Email Routing 已启用：

1. 进入 Cloudflare Dashboard → 选择你的域名
2. 左侧菜单 → **Email** → **Email Routing**
3. 如果提示未启用，点击 **Get started** 完成初始向导

> 后续通过 API 创建的子域名会自动继承 zone 的 Email Routing 设置。

## 常见问题

### Q: 部署成功但网站打不开？

- 检查 Pages 部署是否成功：Cloudflare Dashboard → Pages → 查看最新部署状态
- 如果使用自定义域名，检查 DNS 解析是否正确

### Q: 登录时 OAuth 报错？

- 检查 OAuth 回调地址是否正确：`https://你的域名/api/auth/callback/github`（或 google）
- 确认 `AUTH_GITHUB_ID`、`AUTH_GITHUB_SECRET` 等 Secrets 值正确

### Q: 创建子域名邮箱失败（502）？

1. 检查 `DNS_WORKER_URL` 是否有效（可在浏览器直接访问，应返回 `{"error":"Unauthorized"}`）
2. 检查 API Token 是否有 `Zone > DNS > Edit` 和 `Zone > Email Routing Rules > Edit` 权限
3. 检查管理面板中域名的 Zone ID 是否正确

### Q: DNS Worker 部署失败？

- 确认 `wrangler.dns.example.json` 文件存在
- 确认 API Token 有 `Workers Scripts > Edit` 权限

### Q: 子域名创建成功但收不到邮件？

1. 确认 Email Worker 已正确部署
2. 确认 API Token 有 `Email Routing Rules - Edit` 权限
3. 在 Cloudflare Dashboard → Email Routing → Routing Rules 中检查 catch-all 规则是否指向你的 Email Worker
4. 检查 DNS Worker 的 `EMAIL_WORKER_NAME` Secret 是否正确（应为 `{PROJECT_NAME}-email-receiver-worker`）

### Q: 每次部署都生成新的 DNS_WORKER_SECRET？

如果没有在 GitHub Secrets 中设置 `DNS_WORKER_SECRET`，部署脚本会每次自动生成新密钥。建议首次部署成功后：

1. 查看 GitHub Actions 部署日志，找到生成的 Secret
2. 将其保存到 GitHub Secrets 中的 `DNS_WORKER_SECRET`
3. 同样保存 `DNS_WORKER_URL`

这样后续部署会使用固定值，避免密钥不匹配。

### Q: 如何更新部署？

```bash
# 推送新 Tag 触发自动部署
git tag v1.0.1
git push origin v1.0.1
```

或在 GitHub Actions 页面手动触发 **Run workflow**。
